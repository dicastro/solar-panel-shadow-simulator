Configuration Guide

This guide explains every section of the config.json file used by the Solar Panel Shadow Simulator. You can edit the configuration directly in the app via Settings → Configuration.

1. Site location

The location object sets the GPS coordinates of the installation. These are used to compute the sun's position (altitude and azimuth) for every simulated time step.

"location": {
  "latitude": 40.6252,
  "longitude": -4.0141
}
FieldTypeDescription
latitudenumberDecimal degrees. Positive = North, negative = South. Range −90 to 90.
longitudenumberDecimal degrees. Positive = East, negative = West. Range −180 to 180.

Also set the timezone field to the correct IANA timezone identifier for the installation:

"timezone": "Europe/Madrid"

You can find your timezone identifier at Wikipedia's TZ database list. Common examples: Europe/London, America/New_York, Asia/Tokyo, Australia/Sydney.

2. Site azimuth

The site azimuth rotates the entire installation (walls, railings, panels) in the horizontal plane. It represents the direction the front face of the terrace is pointing, measured in degrees from South.

ValueMeaning
0South-facing (no rotation)
+45Rotated 45° toward East (South-East facing)
-45Rotated 45° toward West (South-West facing)
90East-facing
-90West-facing
180 or -180North-facing
South-facing — azimuth: 0
South-East facing — azimuth: 45

Panel azimuth is absolute, not relative to site azimuth. Each panel array has its own azimuth field inside arrays[], and it uses the same convention (degrees from South). If your site faces South-East (site.azimuth: 45) and you want the panels to also face South-East, set array.azimuth: 45. If you want the panels to face due South regardless of the site orientation, set array.azimuth: 0.

3. Site appearance

These optional fields control the visual appearance of the installation in the 3D view. They have no effect on shadow calculations or energy production estimates.

FieldTypeDefaultDescription
floorColor string (CSS hex) #cccccc Colour of the terrace floor surface. Must be a 6-digit hex colour. Examples: #b45d16 terracotta, #888888 concrete, #cccccc light grey. 
"floorColor": "#b45d16"

4. Installation footprint (wall points)

The wallPoints array defines the perimeter of the terrace as a sequence of corner coordinates in metres. The coordinate system is:

  • +X = East, +Z = North (before any site rotation)
  • The origin [0, 0] can be placed anywhere — what matters are the relative positions
  • Walk the perimeter counter-clockwise starting from the South-West corner

Only 90° and 180° angles are supported. Non-right angles will produce incorrect wall geometry and a warning banner will appear in the app.

Example: rectangular terrace (4m × 6m)

Configuration
"wallPoints": [
  [0, 0],
  [4, 0],
  [4, 6],
  [0, 6]
]
Diagram (top view, North = up)

Red arrows show CCW walk order

Example: L-shaped terrace

An L-shape requires 6 points. All internal angles must be 90°.

Configuration
"wallPoints": [
  [0, 0],
  [6, 0],
  [6, 4],
  [3, 4],
  [3, 8],
  [0, 8]
]
Diagram (top view, North = up)

The SW corner is the minimum X and minimum Z across all points. Panel array positions (arrays[].position) are measured in metres from this SW corner.

5. Wall defaults

The wallDefaults object sets the height and thickness applied to all walls unless overridden.

"wallDefaults": {
  "height": 0.72,
  "thickness": 0.145
}
FieldUnitDescription
heightmetresWall height above the terrace floor.
thicknessmetresWall thickness. Affects both rendering and shadow geometry.

6. Per-wall overrides

Individual walls can override the defaults via wallsSettings. Each entry targets one wall by its 0-based index — the index of the start point of that segment in wallPoints. Wall 0 goes from wallPoints[0] to wallPoints[1], wall 1 from wallPoints[1] to wallPoints[2], and so on.

"wallsSettings": [
  {
    "wall": 2,
    "override": {
      "height": 1.2
    }
  }
]

This raises wall 2 (the third side of the perimeter) to 1.2 m while all other walls stay at the default height. You can override height and/or the railing configuration for any individual wall.

7. Railings

Railings sit on top of walls. The railingDefaults object sets the global railing style; individual walls can override it via wallsSettings[].override.railing.

"railingDefaults": {
  "active": true,
  "heightOffset": 0.185,
  "shape": { "kind": "cylinder", "radius": 0.025 },
  "support": {
    "shape": { "kind": "cylinder", "radius": 0.008 },
    "count": 3,
    "edgeDistance": 0.22
  },
  "extendAtStart": true,
  "extendAtEnd": true,
  "extensionGap": 0.04
}
FieldType / valuesDescription
activebooleanWhether to render a railing on this wall.
heightOffsetmetresDistance from the top of the wall to the centre of the railing rail.
shapesee belowCross-section of the railing rail.
supportsee belowBalusters that hold the rail.
extendAtStart / extendAtEndbooleanWhether the rail extends over the corner post at each end, for a flush corner join.
extensionGapmetresGap between two meeting rail extensions at a corner post.

Rail shapes

Cylinder (round bar)
"shape": {
    "kind": "cylinder",
    "radius": 0.025
  }
Square (flat bar)
"shape": {
    "kind": "square",
    "width": 0.04,
    "height": 0.008
  }

Support (baluster) configuration

FieldDescription
shapeCross-section: cylinder (with radius) or square (with width and depth).
countNumber of balusters per wall. Minimum 2.
edgeDistanceDistance from each wall end to the nearest baluster. When omitted, balusters are distributed uniformly.

8. System-level losses

These three optional fields on site model real-world energy losses between the panels and the grid connection point.

"groundAlbedo": 0.20,
"inverterEfficiency": 0.97,
"wiringLoss": 0.02
FieldDefaultDescription
groundAlbedo0.20Fraction of global horizontal irradiance reflected by the ground toward the panel underside. 0.20 = concrete/asphalt. 0.25 = light gravel. 0.10 = dark roofing membrane.
inverterEfficiency0.97DC-to-AC conversion efficiency of the inverter. Typical modern string inverters: 0.96–0.98.
wiringLoss0.02Fraction of DC power lost in cabling between panels and inverter. Typical residential installations: 0.01–0.03.

9. Setups

A setup is one complete panel configuration to simulate. You can define multiple setups in the setups array to compare different layouts, orientations, or equipment choices side by side.

"setups": [
  { "label": "Current installation", ... },
  { "label": "With optimizers on row 1", ... }
]

Each setup has a label (shown in the UI and in reports), a panelDefaults block, and one or more arrays.

10. Panel defaults

The panelDefaults object defines the physical and electrical characteristics of the panels used in all arrays of this setup, unless overridden at the array level.

"panelDefaults": {
  "width": 1.134,
  "height": 2.094,
  "peakPower": 415,
  "zones": 3,
  "zonesDisposition": "horizontal",
  "hasOptimizer": false,
  "string": "S1",
  "temperatureCoefficient": -0.0036,
  "noct": 44
}
FieldUnit / valuesDescription
widthmetresPanel width as printed on the datasheet.
heightmetresPanel height as printed on the datasheet.
peakPowerWpPeak power at Standard Test Conditions (1000 W/m², 25°C).
zonesinteger ≥ 1Number of bypass-diode zones. Most residential panels have 2 or 3 zones.
zonesDispositionhorizontal / verticalHow zones are split across the panel face. horizontal = stacked top-to-bottom (most common). vertical = side by side.
hasOptimizerbooleanWhether each panel has a DC power optimizer (e.g. SolarEdge). Optimizers remove the string mismatch penalty but add a small 10 % gain cap removal.
stringstringIdentifier of the string this panel belongs to. All panels in the same string share the same series current constraint. Example: "S1", "S2".
temperatureCoefficient/°CRelative change in peak power per degree above 25°C. Found on the datasheet as Pmax temperature coefficient. Typical value: −0.004 /°C. Default when omitted: −0.004.
noct°CNominal Operating Cell Temperature. Found on the datasheet. Used to estimate cell temperature from ambient temperature and irradiance. Default when omitted: 45.

Zones and bypass diodes

A bypass diode protects one zone. When a zone is fully shaded, its diode activates and that zone produces zero power, but the rest of the panel continues. The simulation counts how many sample points in each zone are shaded and applies a configurable threshold to decide whether the zone is considered blocked.

zonesDisposition: "horizontal" is the most common layout — the panel is divided into horizontal bands running across its width. This matches how most panels are physically wired with bypass diodes.

11. Panel arrays

Each setup contains one or more arrays. An array is a rectangular grid of identical panels at a fixed position, inclination, and azimuth.

"arrays": [
  {
    "position": [1.5, 1.0],
    "azimuth": 0,
    "elevation": 0.10,
    "inclination": 20,
    "rows": 3,
    "columns": 2,
    "spacing": [0.02, 0.04],
    "orientation": "portrait"
  }
]
FieldUnit / valuesDescription
position[East metres, North metres]Distance from the site SW corner to the SW corner of this array.
azimuthdegreesAbsolute panel azimuth (0 = South, +East, −West). Independent of site azimuth.
elevationmetresHeight of the bottom edge of the array above the terrace floor.
inclinationdegreesTilt angle from horizontal. 0° = flat. 90° = vertical. Typical residential: 10–35°.
rowsintegerNumber of panel rows. Row 0 is the southernmost; rows increase northward.
columnsintegerNumber of panel columns. Column 0 is the westernmost; columns increase eastward.
spacing[horizontal m, vertical m]Gap between adjacent panels. Optional; defaults to [0.02, 0.02].
orientationportrait / landscapeWhether panels are mounted taller-than-wide or wider-than-tall. Optional; defaults to portrait.

Array-level overrides — width, height, peakPower, zones, zonesDisposition, hasOptimizer, string, temperatureCoefficient, noct — can all be set directly on the array object to override the setup's panelDefaults.

Example: two arrays on different strings

"arrays": [
  {
    "position": [0.5, 1.0],
    "azimuth": 0, "elevation": 0.10, "inclination": 18,
    "rows": 2, "columns": 3,
    "string": "S1"
  },
  {
    "position": [0.5, 4.5],
    "azimuth": 0, "elevation": 0.10, "inclination": 18,
    "rows": 1, "columns": 3,
    "string": "S2"
  }
]

12. Per-panel overrides

Individual panels within an array can be overridden via arraysSettings. Each entry targets one panel by its array index, row, and column.

"arraysSettings": [
  {
    "array": 0,
    "row": 0,
    "col": 2,
    "hasOptimizer": true,
    "string": "S2"
  }
]

This moves the panel at array 0, row 0, column 2 to string S2 and adds an optimizer to it. All other properties are inherited from the array's defaults. Multiple entries can be listed to override several panels.

Panels with optimizers are shown in green in the 3D view; panels without optimizers are shown in dark blue.

Each string is assigned a distinct colour automatically. A colour-coded border is rendered on the front face of each panel in the 3D view, on the heat maps in the results panel, and in PDF reports. The colour is consistent within a setup and assigned by first-appearance order of string identifiers. Up to 10 strings can be visually distinguished; beyond that, colours repeat.

13. Full example

A complete configuration for a real-world style installation: rectangular 5m × 8m terrace in Madrid, one panel array of 2 rows × 4 columns, South-facing, 20° inclination, 415 Wp panels with 3 horizontal bypass zones.

{

    "location": { "latitude": 40.4168, "longitude": -3.7038 },
    "azimuth": 0,
    "timezone": "Europe/Madrid",
    "groundAlbedo": 0.20,
    "inverterEfficiency": 0.97,
    "wiringLoss": 0.02,
    "wallPoints": [
      [0, 0], [5, 0], [5, 8], [0, 8]
    ],
    "wallDefaults": { "height": 0.72, "thickness": 0.145 },
    "railingDefaults": {
      "active": true,
      "heightOffset": 0.185,

    {
      "label": "South 20deg 2x4",
