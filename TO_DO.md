* La "fase 5" se refiere a tener en cuenta la irradiancia proveniente de PVGIS. Actualmente ya hay un componente en la UI para elegir la fuente de la irradiancia, permitiendo valores: geométrica, PVGIS y Open-Meteo. En la "fase 5" se menciona de tener en cuenta PVGIS. Si el approach puede ser igual para PVGIS y para Open-Meteo, desarrolla las 2 opciones. Si PVGIS y Open-Meteo funcionan totalmente diferente, o alguna de ellas no devuelve los valores que nos interesa, o el API necesita un montón de llamadas y lo hace inviable, descarta la opción complicada e implementa la que sea factible. La idea es realizar el menor número de requests http para recuperar la información de la irradiancia horaria (o para el intervalo seleccionado, habría que ver cuál es el intervalo de tiempo mínimo) para el año seleccionado. Si por lo que sea el usuario ha seleccionado un intervalo menor a 1h y el API de la irradiancia no tiene una resolución menor a 1h, se utilizará para todos los intervalos el mismo valor de irradiancia. En este caso lo ideal es que los valores del combo de "Intervalo" estuvieran condicionados a la fuente de "Irradiancia" seleccionada. Si PVGIS por ejemplo lo limita a 1h, no tiene sentido elegir un intervalo de 15min. Pero sí tendría sentido el intervalo de 15min si se elige la opción de irradiancia "Geometrica". Tal y como se menciona, la información de irradiancia debería ser cacheada para no volver a hacer la llamada. No sé cómo funcionan estas APIs, pero la lógica me dice que si pido datos para un año del pasado, los datos devueltos van a ser constantes y siempre van a ser iguales, porque se trata de un tiempo pasado. Sin embargo, si pido los datos para el año en curso, entiendo que el futuro será una predicción. Así que igual solo tiene sentido cachear estos datos para años pasados y para el año en curso (con datos futuros) tiene sentido cachearlos en base a cuándo se pidieron, por ejemplo que se cacheen todo el día, si los pido hoy varias veces para el año en curso, serán siempre iguales durante el día de hoy, pero si los pido mañana podrían ser diferentes. A la hora de cachear los del año en curso, solo quiero que se cachee una versión, la última información recuperada, no quiero que esa caché pueda crecer infinitamente. Quiero que cada integración con una fuente de irradiancia esté en su propio módulo, para que esté organizado y para que si el día de mañana se añade otra, no cueste mucho. Si alguna de estas apis requiriese de un registro y uso de tokens, quedaría descartada. Tiene que ser algo sencillo y que cualquier usuario que llegue a mi aplicación la pueda usar sin tener que registrarse en ningún sitio.
